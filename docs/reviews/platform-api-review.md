# Platform API Review — Factual Accuracy of the instagram-mcp Documentation Set

## 1. Reviewer & scope

- **Role:** Senior Meta Platform integration specialist (Instagram Platform / Graph API).
- **Docs reviewed:** `README.md`, `docs/auth.md`, `docs/tools.md`, `docs/operations.md` (primary focus), plus `docs/architecture.md`, `docs/security.md`, `docs/roadmap.md`.
- **Background corpus:** `facebook-mcp/docs/ai/research/meta-auth-permissions.md` (vetted 2026-07-21).
- **Date:** 2026-07-21. Docs assume Graph API **v25.0** current (released 2026-02-18) — consistent with the official changelog.
- **Verification method:** Every **[verify]** marker plus additional doubtful claims checked by fetching **official Meta documentation** at `developers.facebook.com` (both the legacy `/docs/` tree and the new `/documentation/` tree), supplemented by targeted web search only to locate the canonical pages or where official pages are silent (flagged where so). No reviewed document was modified.

## 2. Executive summary

**Verdict: CONDITIONAL GO** on API-factual grounds. The documentation set is unusually accurate for a design-phase corpus: the dual-path auth model, token lifecycle, two-phase publishing flow, comment/mention endpoints, post-2025 insights metric set, hashtag budget, and versioning strategy are all essentially correct, and every genuinely uncertain claim was correctly flagged with **[verify]**. The design's instinct to distrust its own numbers is validated — several of the flagged items did need correction.

Most important corrections (details in §4):

1. **`media_type=IMAGE` does not exist.** Image containers are created *without* a `media_type` parameter; valid values are only `REELS`, `STORIES`, `CAROUSEL` (`VIDEO` is likewise not a valid value — feed video *is* Reels). (F-1)
2. **JPEG is the only supported image format** — the "JPEG/PNG" client-side validation spec would accept files Meta rejects. (F-2)
3. **Reels max file size is 300 MB**, not ≤ 1 GB (stories video: 100 MB / 60 s). (F-3)
4. **Publishing quota: Meta's own docs disagree** (guide: 100/24 h; reference + carousel section: 50/24 h). The server must treat `config.quota_total` from `content_publishing_limit` as authoritative at runtime, never a hardcoded constant. (F-4)
5. **Hashtag media requires the "Instagram Public Content Access" feature**, which is an App-Review-gated feature — this dents the "no App Review ever" narrative for the `discovery` package. (F-5)
6. **Instagram BUC rate limit is a rolling 24-hour window** (`4800 × impressions`), not 1 hour, and its throttle error code is **80002** — missing from the error taxonomy. (F-6)
7. **Container-expired error is code 24 / subcode 2207008**, not 9007 (9007/2207027 = "media not ready"); publishing-quota-reached is **code 9 / subcode 2207042**; **subcode 2207051 is a spam/integrity restriction and must not be auto-retried**. (F-7, F-8)
8. Meaningful platform capabilities added in late 2025 / 2026 are missing from the catalog: **`DELETE /{ig-media-id}`**, **`alt_text`**, **`collaborators`**, **`is_ai_generated`** (highly relevant for an AI-driven server), resumable video upload (lifts the public-URL constraint for video), tokenless oEmbed. (§5)

Confirmed intact: all Path-A scope names, `ig_exchange_token`/`ig_refresh_token` (60 days, 24 h minimum age), `graph.instagram.com` versioned `/v25.0/` paths, Path-A lack of hashtag search / business discovery / product tagging, container status codes and 24 h expiry, caption 2,200 chars / 30 hashtags / 20 @tags, 30-unique-hashtags-per-7-days, comments/mentions endpoints, and the post-2025 insights metric set.

## 4. Findings

### F-1 · High · tools.md §publishing (`instagram_create_media_container`)
`media_type=IMAGE` is specified as the default enum value, but **`IMAGE` is not a valid `media_type`**. The Graph call as designed would fail on the single most common operation. Correct behavior: omit `media_type` entirely for single feed images (presence of `image_url` implies an image container); send `media_type` only for `REELS`, `STORIES`, `CAROUSEL` (and `is_carousel_item=true` on carousel children). `VIDEO` must not be offered either — feed video is Reels. The tool's zod schema should model this as a discriminated union (`kind: image|reel|story|carousel`) that maps to the correct parameter set, rather than exposing Meta's enum raw.
Source: https://developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-user/media/

### F-2 · High · tools.md §publishing design note
"Image spec (JPEG/PNG 8 MB…)" — **PNG is not supported**; "JPEG is the only image format supported" (MPO/JPS extended JPEG also rejected). The client-side pre-validation would happily pass PNGs that then die in the container with an opaque error. Fix the validation spec to JPEG-only and document the constraint to operators (it routinely surprises people). Aspect ratio 4:5–1.91:1 and 8 MB are correct; add width bounds 320–1440 px (Meta auto-scales outside this range).
Source: https://developers.facebook.com/docs/instagram-platform/content-publishing/

### F-3 · High · tools.md §publishing design note
Reels spec "3 s–15 min, ≤ 1 GB" — duration correct, but the documented max file size is **300 MB**. Also missing: stories video is capped at **60 s / 100 MB**, stories images at 8 MB JPEG (9:16 recommended). The 1 GB validation bound would wave through files three times the real limit.
Source: https://developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-user/media/

### F-4 · High · tools.md `instagram_get_publishing_limit`, operations.md §1, security.md §4
The "100 posts / 24 h" figure appears in the current guide's rate-limit section, but Meta's own documentation is internally inconsistent: the same guide's carousel section says "Accounts are limited to 50 published posts within a 24-hour period," and the `content_publishing_limit` reference states `quota_total` is currently **50**. Since the design already calls the endpoint before composite posts, make that the single source of truth: **never hardcode 100 (or 50)** — read `config.quota_total`/`quota_duration` and compute headroom from `quota_usage`. Docs should say "quota as reported by `content_publishing_limit` (currently 50–100 depending on account/doc version)". Note also the `since` parameter accepts at most 24 h back.
Sources: https://developers.facebook.com/docs/instagram-platform/content-publishing/ , https://developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-user/content_publishing_limit

### F-5 · High · auth.md §2 (App Review reality), tools.md §discovery, roadmap M4
The hashtag media edges require the **"Instagram Public Content Access" feature** in addition to `instagram_basic` ("`Instagram Public Content Access` feature approval" listed under top_media requirements). Features are approval-gated separately from permissions, and public-content features have historically required App Review even for apps whose *permissions* run happily at Standard Access — because the data accessed belongs to *other* accounts, the "own assets, app role" shortcut does not obviously apply. The blanket statement "a solo operator … needs no App Review" is therefore too strong for the `discovery` package (hashtag media, and possibly business_discovery). Action: test at M1 whether hashtag/top_media works for an app-role user without the feature; if not, document the feature request as a prerequisite of the `discovery` package, or mark those tools "may require App Review".
Source: https://developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-hashtag/top-media

### F-6 · Medium · operations.md §1 (rate-limit table), architecture.md §5
Two corrections: (a) the Instagram BUC window is **rolling 24 hours**, not 1 hour — formula "Calls within 24 hours = 4800 × Number of Impressions" (impressions of the account's content in the last 24 h). For small/new accounts this budget is brutally small — worth a doc warning. (b) The BUC throttle surfaces as **error code 80002** for Instagram Platform endpoints — absent from the taxonomy; add it to the retryable rate-limit set alongside 4/17/32/613/429, and honor `estimated_time_to_regain_access` from the BUC header as the reset hint. Also note: no usage headers are documented for `graph.instagram.com` specifically — parse both headers opportunistically and tolerate their absence on Path A.
Source: https://developers.facebook.com/docs/graph-api/overview/rate-limiting

### F-7 · Medium · operations.md §1/§3
Subcode **2207051** is not a benign "publishing user throttled" signal: it is code 4 + "We restrict certain activity to protect our community" — Instagram's spam/integrity restriction on publishing. Mapping it to a retryable `RateLimitError` is actively harmful (retrying reinforces the spam signal). Map it to a distinct non-retryable `RestrictedError` with operator guidance. The plain publishing-quota error is **code 9 / subcode 2207042** — add it, mapped to the quota message with the reset derivable from `content_publishing_limit`.
Source: https://developers.facebook.com/docs/instagram-api/reference/error-codes/

### F-8 · Medium · operations.md §3 (error taxonomy)
"9007 / container `EXPIRED`" conflates two cases: **9007 / 2207027 = media not ready** ("wait until status is FINISHED" — retry by polling, not re-creating), whereas an expired/nonexistent container is **24 / 2207008** (and −2 / 2207020 for expired uploads) — those need re-creation. Getting this backwards would re-create containers that only needed another poll (burning the 400/24 h container budget) or poll dead containers forever. Also add **36003 / 2207009** (invalid aspect ratio) and **352 / 2207026** (unsupported video format) to the `ValidationError` enrichment.
Source: https://developers.facebook.com/docs/instagram-api/reference/error-codes/

### F-9 · Medium · tools.md `instagram_list_mentions`
The tool conflates three distinct surfaces: (a) `GET /{ig-id}/tags` — media the account is **tagged** in (photo tags); (b) `mentioned_media` / `mentioned_comment` — media/comments where the account is **@mentioned** (Path B fields, queried per-ID from webhook or comment data); (c) Path A's `POST /{ig-id}/mentions` — replying to a mention. Recommend splitting into `instagram_list_tagged_media` (both paths, `/tags`) and a mention-reply tool, and documenting that pull-based @mention *discovery* on Path B rides `mentioned_*` lookups while Path A pairs `/tags` with webhooks (out of scope v1). Also: story mentions are not supported on Path A.
Sources: https://developers.facebook.com/docs/instagram-platform/instagram-api-with-instagram-login/mentions/ , https://developers.facebook.com/docs/instagram-platform/instagram-graph-api/reference/ig-user/mentioned_media/

### F-10 · Medium · tools.md §insights
Three gaps: (a) demographics metrics require the **`timeframe`** parameter (`this_week`, `this_month`, `last_14_days`, `last_30_days`, `last_90_days`, `prev_month`) — absent from the tool input; (b) story insights: metrics are only retrievable for the story's **24-hour life**, and EU/Japan viewers are excluded from `replies` — worth stating in the tool description so the model doesn't misread zeros; (c) `total_comments`/`total_likes`/`total_views` (aggregated incl. boosted) and the story `link_clicks` metric are **Facebook-Login-only** — feed these into the M4 auth-path capability matrix.

## 5. Domain recommendations

### 5.1 Missing capabilities worth adding (all verified available)

1. **`is_ai_generated` on publish** (2026-06-22, both login paths): lets the operator attach Instagram's AI-info label at publish time. For an MCP server whose captions/images will often be AI-produced, exposing this flag (and defaulting the preview to surface it) is both a compliance story and a differentiator. https://developers.facebook.com/docs/instagram-platform/changelog
2. **`DELETE /{ig-media-id}`** (2025-12-03, `instagram_manage_contents`): media deletion now exists. Fits the existing double-gate (`apply` + `IG_ALLOW_DESTRUCTIVE`) exactly; the catalog currently has no media-delete tool and the docs predate the capability.
3. **`alt_text` write parameter** (≤ 1,000 chars) on image containers — accessibility + SEO, trivial to add, and a natural fit for an LLM client that can draft alt text. Readable back via the media field.
4. **`collaborators` (≤ 3 usernames)** on containers, plus the collab-invite accept/decline endpoints (2025-12-03) and collaborative-media listing (2026-04-22) — collab posts are core creator workflow now.
5. **Resumable video upload (`upload_type=resumable` via `rupload.facebook.com`)** — the architecture already allowlists `rupload.facebook.com` but the tools doc's "publishing a local file requires a publicly reachable URL" is only true for *images*. Supporting resumable upload for Reels/story video removes the biggest operator pain point without any tunnel/hosting compromise. Strongly recommended for M2 or M2.5.
6. **oEmbed tool** — since 2026-05-15 the Instagram oEmbed API needs **no access token**; a read-only `instagram_get_oembed` (embed HTML/metadata for any public post URL) is nearly free to build and useful for content-reference workflows. Mind the November 2025 field removals (`author_name`, `thumbnail_url`, …). https://developers.facebook.com/docs/instagram-platform/changelog
7. **Like media/comments API** (2026-04-22, `instagram_manage_engagement`) — optional engagement package; low priority but note it for the roadmap's parked list.
8. Publishing extras now available: `audio_name` (reels), `trial_params` (Trial Reels, 2025-12-03), `branded_content_sponsor_ids`/`is_paid_partnership` (2026-04-22), `product_tags` (≤ 5; Path B + approved shop + `instagram_shopping_tag_products` — the guide's "Shopping tags are not supported" line conflicts with the product-tagging surface; treat product tagging as its own gated feature).

### 5.2 Auth-path characterization & primary-path recommendation

The docs' capability characterization is correct and now fully confirmable: Path A (Instagram Login, `graph.instagram.com`) supports publishing, comments, mentions, **insights (since 2025-01-21)**, and native messaging; it lacks **hashtag search, business discovery, product tagging, partnership ads, story-insights webhooks, and the `total_*` aggregate metrics**. Path B has everything plus never-expiring system-user tokens and `appsecret_proof`/`debug_token`.

**Recommendation: make Path A the primary documented path.** Rationale: (a) it covers 100 % of the default `core` profile (account, media, publishing, comments, insights); (b) its setup cost is a fraction of Path B's (no Page link, no Business Manager, and the App Dashboard hands out 60-day long-lived tokens directly — worth documenting as the zero-OAuth quickstart); (c) the target persona is a solo creator/operator. Keep Path B as the documented "power path" required for the `discovery` package, product tags, and set-and-forget token longevity. The dual-path design itself is sound and matches how Meta now positions the platform; the registry-enforced per-path capability matrix planned for M4 is exactly right — build it from the overview comparison table (V9–V11) plus the insights nuances in F-10.

### 5.3 Deprecation watch-list

- **`online_followers`** — alive but absent from the new-tree reference's metric list; re-verify each version bump (V31).
- **Publishing quota value** — 50 vs 100 doc conflict (F-4); watch for the reference catching up.
- **oEmbed** — post-Nov-2025 field set only; legacy fields gone.
- **Graph API v26** — unannounced; expect ~annual cadence. v25 itself contained no Instagram-specific breaks (V41).
- **Docs-tree migration** (`/docs/` → `/documentation/`) — live and messy: several reference pages 404 on one tree and exist on the other (encountered repeatedly during this review). Pin canonical URLs at implementation time, as operations.md already plans.
- **Legacy metric aliases** (`impressions`, `plays`, `clips_replays_count`, `ig_reels_aggregated_all_plays_count`) — fully dead since 2025-04-21; never expose them even as input aliases.

## 6. Verdict

**CONDITIONAL GO** — the documentation set is factually sound at its core; nothing found invalidates the architecture, the dual-path auth design, the write-safety model, or the tool surface. Conditions to clear before/during implementation:

1. Fix the container-creation contract: no `IMAGE`/`VIDEO` media_type; JPEG-only images; 300 MB reels / 100 MB story video bounds (F-1..F-3).
2. Derive the publishing quota from `content_publishing_limit.config` at runtime; never hardcode (F-4).
3. Rework the error taxonomy: 24/2207008 vs 9007/2207027; 9/2207042; 2207051 as non-retryable integrity restriction; add 80002; BUC = 24 h window (F-6..F-8).
4. Resolve the "Instagram Public Content Access" feature question empirically at M1 and re-scope the App-Review claim for the `discovery` package (F-5).
5. Split/clarify the mentions tooling and add `timeframe` to demographics (F-9, F-10).
6. Adopt at minimum `is_ai_generated` and `alt_text` in the publishing schema; decide on resumable video upload for M2 (§5.1).

With those corrections folded in, the design is fit to proceed to M0/M1.
